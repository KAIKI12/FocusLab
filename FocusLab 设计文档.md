## 1. 产品概述

### 1.1 产品定位

FocusLab 是一款面向在校研究生的桌面端专注力与任务管理应用。它不是一个通用的待办清单工具，而是一个围绕研究生科研节奏设计的「个人工作操作系统」。

核心理念：**陪伴式效率管理** —— 不做监工，做镜子；不做法官，做伙伴。

### 1.2 要解决的核心问题

| 问题编号 | 痛点描述 | 现有工具的不足 |
|---------|---------|--------------|
| P1 | 研究生的任务颗粒度模糊，"写论文"太大，不知道今天该做什么 | 通用待办工具不提供任务拆解引导 |
| P2 | 长线任务（如发表论文）缺乏阶段感，容易焦虑或麻木 | 没有里程碑式的长线进度追踪 |
| P3 | 自我管理能力参差不齐，缺乏外部反馈 | 现有工具只记录，不反馈 |
| P4 | 不知道时间花在了哪里，缺乏自我觉察 | 计时工具与任务管理割裂 |
| P5 | 未完成任务带来心理负担，导致逃避甚至卸载工具 | 现有工具用红色标记未完成，加重焦虑 |

### 1.3 产品差异化

```
市面上的效率工具：
├── 滴答清单 / Todoist → 通用待办，无科研场景适配
├── Forest → 只有专注计时，无任务管理
├── Notion → 过于灵活，需要用户自己搭建系统
├── 番茄Todo → 番茄钟+待办，但无长线目标追踪
└── 以上所有 → 不提供引导思考、AI拆解、情绪感知

FocusLab 的差异化：
├── ✅ 专为研究生科研节奏设计
├── ✅ 引导式每日计划（帮用户想清楚今天做什么）
├── ✅ 长线目标 + 里程碑 + 每日小目标的三层映射
├── ✅ AI 驱动的任务拆解与个性化建议
├── ✅ 温和的未完成任务处理机制（不制造焦虑）
└── ✅ 游戏化日结算反馈（但不是游戏）
```

## 2. 目标用户分析

### 2.1 核心用户画像

**主要用户：在校研究生（硕士/博士）**

```
👤 用户画像 A：硕士研究生小李
├── 年龄：23岁，研二
├── 日常：每天在实验室工位工作 8-10 小时
├── 痛点：
│   ├── 导师给了大方向，但不知道每天具体做什么
│   ├── 论文写了两个月，感觉没什么进展
│   ├── 经常被事务性工作打断（报销、填表、开会）
│   └── 晚上回去觉得"今天好像什么都没做"
├── 期望：
│   ├── 有人帮我理清今天该做什么
│   ├── 能看到自己的长期进展
│   └── 不要给我太大压力
└── 技术水平：会用基本软件，不排斥新工具

👤 用户画像 B：博士研究生小王
├── 年龄：27岁，博三
├── 日常：同时推进论文、项目、助教工作
├── 痛点：
│   ├── 多条线并行，经常顾此失彼
│   ├── 长线任务太多，不知道优先级
│   ├── 时间碎片化严重
│   └── 焦虑感强，效率工具反而加重焦虑
├── 期望：
│   ├── 帮我管理多条并行任务线
│   ├── 告诉我时间花在了哪里
│   └── 不要让我觉得自己很失败
└── 技术水平：熟悉各种工具，对产品有较高要求
```

### 2.2 用户使用场景

```
场景1：早晨到工位
├── 打开电脑，FocusLab 自动启动
├── 显示昨日复盘卡片 + 今日引导
├── 用户花 3 分钟设定今日计划
└── 开始第一个番茄钟

场景2：工作中
├── FocusLab 以迷你悬浮球形式常驻
├── 显示当前任务名 + 番茄钟倒计时
├── 番茄钟结束时，柔和提醒休息
└── 用户切换任务时，一键切换计时

场景3：被打断
├── 导师突然叫去开会
├── 用户点击"暂停"，记录中断原因
├── 回来后点击"继续"或切换到新任务
└── 中断数据被记录，用于分析时间碎片化

场景4：一天结束
├── 用户点击"结束今天"或到达设定的结算时间
├── 弹出日结算面板，展示完成情况
├── 根据完成度给予分级反馈
└── 用户可以写一句今日感想（可选）

场景5：周末回顾
├── 用户打开周报功能
├── 查看本周时间分布、完成率趋势
├── AI 生成周度总结和建议
└── 用户调整下周的长线目标计划
```

### 2.3 用户心理模型

```
研究生使用效率工具的心理周期：

第1天：兴奋期 → "这个工具好酷，我要好好用"
第3天：适应期 → "还行，但有些功能用不上"
第7天：倦怠期 → "每天打开有点烦，但还在用"
第14天：分水岭 → 要么形成习惯，要么开始遗忘
第30天：稳定期 → 如果还在用，说明真的有价值

FocusLab 的留存策略：
├── 第1-3天：简化流程，不要让用户觉得复杂
├── 第3-7天：开始展示数据洞察，让用户看到价值
├── 第7-14天：引入长线目标追踪，建立沉没成本
├── 第14-30天：AI 建议开始基于历史数据，越来越精准
└── 第30天+：用户已经积累了大量数据，迁移成本高
```

## 3. 产品设计原则

### 3.1 核心原则

```
原则1：陪伴，不监工
├── 不用红色标记未完成任务
├── 不用"失败""落后"等负面词汇
├── 用"搁置""调整""重新开始"替代
└── 让用户感到被支持，而不是被评判

原则2：镜子，不法官
├── AI 呈现数据事实，不做价值判断
├── "你今天在XX上花了3小时" ✅
├── "你今天效率太低了" ❌
└── 让用户自己得出结论

原则3：工具，不游戏
├── 不做经验值、等级、排行榜
├── 成就系统奖励"坚持"而非"完成量"
├── 视觉反馈要克制，不要过度花哨
└── 核心价值是帮用户管理时间，不是娱乐

原则4：渐进，不强制
├── 所有引导流程都可以跳过
├── 功能复杂度随使用时间逐步展开
├── 新用户看到简洁界面，老用户解锁高级功能
└── 永远给用户"不用这个功能"的选项

原则5：轻量，不打扰
├── 常驻时占用最小屏幕空间
├── 提醒用柔和方式（呼吸灯效果，不是弹窗）
├── 尊重用户的专注状态
└── 用户在深度工作时，应用应该"隐形"
```

### 3.2 情感设计准则

```
语言风格：
├── 温暖但不油腻："今天辛苦了" ✅ "你是最棒的!!!" ❌
├── 诚实但不刺耳："今天完成了3个任务" ✅ "你只完成了3个" ❌
├── 建议但不命令："可以试试..." ✅ "你应该..." ❌
└── 用户可自定义AI语气风格

视觉风格：
├── 主色调：柔和的蓝绿色系（专注、平静）
├── 成功反馈：暖色光效（金色、橙色）
├── 未完成：灰色淡化（不是红色警告）
├── 整体氛围：像一个安静的图书馆，不是嘈杂的游戏厅
└── 支持深色模式（研究生经常熬夜）
```

## 5. 核心功能详细设计

### 5.1 专注计时模块

#### 5.1.1 番茄钟模式

```
功能描述：
基于番茄工作法的计时系统，支持多种时长模式，
与任务系统深度集成。

计时模式：
├── 🍅 经典模式：25分钟专注 + 5分钟休息
├── 🍅🍅 深度模式：45分钟专注 + 10分钟休息
├── 🍅🍅🍅 沉浸模式：90分钟专注 + 15分钟休息
└── ⏱️ 自由模式：不设时限，手动开始/结束

长休息规则：
└── 每完成 4 个番茄钟，触发一次长休息（15-30分钟）

计时状态机：
┌──────┐    开始    ┌──────┐   时间到   ┌──────┐
│ 空闲  │ ────────→ │ 专注中 │ ────────→ │ 休息中 │
└──────┘           └──────┘           └──────┘
                     │  ↑                │
                暂停  │  │ 继续           │ 休息结束
                     ↓  │                ↓
                   ┌──────┐          ┌──────┐
                   │ 暂停中 │          │ 空闲  │
                   └──────┘          └──────┘
                     │
                放弃  │
                     ↓
                   ┌──────┐
                   │ 已放弃 │ → 记录放弃原因（可选）
                   └──────┘
```

#### 5.1.2 中断与切换

```
中断处理：
├── 用户点击"暂停"时，弹出中断原因选择：
│   ○ 🧑‍🏫 导师/同学找
│   ○ 📞 电话/消息
│   ○ 🚶 主动休息
│   ○ 🔄 切换任务
│   ○ 📝 其他
├── 中断数据被记录，用于分析时间碎片化程度
└── 中断原因选择是可选的，用户可以直接暂停不填

任务切换：
├── 在专注中点击"切换任务"
├── 当前任务自动暂停并记录已用时间
├── 弹出任务列表，选择新任务
├── 新任务开始计时
└── 切换记录保存，用于分析多任务切换频率

回溯补录：
├── 用户忘记按"开始"时，可以事后补录
├── 选择任务 → 输入开始时间和结束时间
├── 补录的时间段标记为"手动补录"
└── 补录数据参与统计，但在分析中标注来源
```

#### 5.1.3 计时数据记录

```
每个专注时段记录以下数据：
{
  session_id: 唯一标识,
  task_id: 关联的任务,
  start_time: 开始时间,
  end_time: 结束时间,
  planned_duration: 计划时长（番茄钟模式下）,
  actual_duration: 实际时长,
  mode: "pomodoro" | "free",
  status: "completed" | "paused" | "abandoned",
  interruptions: [
    { time: 中断时间, reason: 中断原因, duration: 中断时长 }
  ],
  is_manual_entry: 是否手动补录
}
```

### 5.2 任务管理模块

#### 5.2.1 任务创建

```
任务基本属性：
├── 任务名称（必填）
├── 任务描述（可选）
├── 所属分类（科研核心/事务性/学习提升/日常例行/个人发展）
├── 四象限位置（紧急重要/不紧急重要/紧急不重要/不紧急不重要）
├── 预估时长（可选，AI可辅助预估）
├── 关联长线目标（可选）
├── 关联里程碑（可选）
├── 截止日期（可选）
├── 重复规则（可选：每天/每周/自定义）
└── 标签（自定义标签，如"论文A""项目B"）

任务创建方式：
├── 方式1：手动创建（填写表单）
├── 方式2：快速创建（只输入名称，其他后补）
├── 方式3：AI拆解创建（输入大任务，AI拆解为子任务）
├── 方式4：从昨日遗留中继承
└── 方式5：从引导思考流程中生成
```

#### 5.2.2 四象限分类系统

```
四象限矩阵：

              紧急                    不紧急
        ┌──────────────────┬──────────────────┐
        │                  │                  │
  重要  │  🔴 第一象限      │  🟡 第二象限      │
        │  立即执行         │  计划执行         │
        │                  │                  │
        │  例：明天DDL的报告│  例：论文写作      │
        │  导师今天要的数据 │  学习新方法       │
        │                  │                  │
        ├──────────────────┼──────────────────┤
        │                  │                  │
 不重要 │  🟠 第三象限      │  🟢 第四象限      │
        │  委托或快速处理   │  减少或搁置       │
        │                  │                  │
        │  例：回复非紧急邮件│  例：整理桌面     │
        │  填写报销单       │  刷学术八卦       │
        │                  │                  │
        └──────────────────┴──────────────────┘

AI 辅助分类：
├── 用户创建任务时，AI 根据任务描述建议象限位置
├── 用户可以接受或手动调整
├── AI 学习用户的分类习惯，建议越来越准确
└── 每日计划时，AI 建议优先处理第一象限，然后第二象限
```

#### 5.2.3 任务状态流转

```
任务生命周期：

  ┌──────┐   开始    ┌──────┐   完成    ┌──────┐
  │ 待开始 │ ───────→ │ 进行中 │ ───────→ │ 已完成 │
  └──────┘          └──────┘          └──────┘
     │                 │
     │ 搁置            │ 搁置
     ↓                 ↓
  ┌──────┐          ┌──────┐
  │ 已搁置 │ ←────── │ 已搁置 │
  └──────┘          └──────┘
     │
     │ 恢复
     ↓
  ┌──────┐
  │ 待开始 │ （重新进入队列）
  └──────┘

搁置规则：
├── 用户可以主动搁置任何任务
├── 连续3天未处理的遗留任务，系统建议搁置
├── 搁置时可选填原因（任务太难/优先级变了/不想做了）
```

### 5.4 日结算模块

#### 5.4.1 结算触发机制

```
触发方式（多种方式并存）：

方式A：手动触发（推荐）
├── 用户点击"结束今天"按钮
└── 适用于：用户主动结束工作日

方式B：智能触发
├── 检测到用户结束最后一个任务后 10 分钟无操作
├── 弹出提示："今天的工作结束了吗？要进行日结算吗？"
└── 适用于：用户忘记手动触发

方式C：定时触发（兜底）
├── 用户在设置中自定义结算时间（默认 22:00）
├── 到达时间后弹出提醒
├── 如果用户忽略，不再重复弹出（第二天早上再展示）
└── 适用于：确保每天都有结算

优先级：方式A > 方式B > 方式C
```

#### 5.4.2 结算分级系统

```
分级规则：

S级 - 超额完成（完成率 ≥ 120%，即完成了额外任务）
├── 判定条件：所有计划任务完成 + 额外完成了非计划任务
├── 视觉效果：金色光效 + 彩虹渐变进度条 + 粒子特效
├── 音效：成就解锁音效（可关闭）
├── 进度条样式：彩虹色，带流光动画
└── AI 文案风格：热烈祝贺 + 数据亮点

A级 - 完美完成（完成率 = 100%）
├── 判定条件：所有计划任务全部完成
├── 视觉效果：发光进度条 + 星星特效
├── 音效：清脆的完成音
├── 进度条样式：金色，带发光效果
└── AI 文案风格：肯定 + 明日展望

B级 - 基本完成（完成率 70% - 99%）
├── 判定条件：大部分任务完成，少量未完成
├── 视觉效果：蓝色进度条，柔和
├── 音效：柔和提示音
├── 进度条样式：蓝色渐变
└── AI 文案风格：肯定已完成的 + 温和提及未完成

C级 - 需要调整（完成率 < 70%）
├── 判定条件：较多任务未完成
├── 视觉效果：灰蓝色进度条（不用红色）
├── 音效：无音效或极轻柔提示
├── 进度条样式：灰蓝色，不带动画
└── AI 文案风格：理解 + 建议调整 + 鼓励明天重新开始

完成率计算方式：
├── 基础完成率 = 已完成任务数 / 计划任务总数 × 100%
├── 超额部分 = 额外完成的非计划任务数 / 计划任务总数 × 100%
├── 最终完成率 = 基础完成率 + 超额部分
└── 注意：搁置的任务不计入分母
```

### 5.5 长线目标模块

#### 5.5.1 长线目标结构

```
长线目标的三层结构：

第一层：长线目标（Goal）
├── 例：发表一篇SCI论文
├── 属性：名称、描述、预计完成时间、状态
└── 一个用户可以同时有多个长线目标

第二层：里程碑（Milestone）
├── 例：完成文献综述、完成方法论设计
├── 属性：名称、描述、预计完成时间、状态、排序
├── 每个长线目标包含多个里程碑
└── 里程碑之间有先后顺序

第三层：子任务（Sub-task）
├── 例：阅读论文A、复现baseline代码
├── 属性：与普通任务相同
├── 每个里程碑包含多个子任务
└── 子任务可以被拉入每日计划

映射关系：
长线目标 1:N 里程碑 1:N 子任务 → 每日任务

用户每天设定的任务可以关联到某个里程碑的子任务，
从而建立"今天做的事"与"长期目标"之间的清晰联系。
```

#### 5.5.3 长线目标与每日任务的关联

```
关联方式：

1. 创建任务时关联
├── 用户创建任务时，可选择"关联到长线目标"
├── 选择目标 → 选择里程碑 → 任务自动成为该里程碑的子任务
└── 也可以不关联（独立任务）

├── 引导流程的Step 2会展示长线目标进度

├── 系统自动展示论文A当前里程碑的待完成子任务

3. AI建议时关联
├── AI分析长线目标进度后
├── 建议用户今天应该推进哪个目标的哪个子任务
└── 用户采纳后自动关联

数据价值：
├── 用户可以看到"今天做的事对长期目标的贡献"
├── 周报中展示"本周在各长线目标上的时间投入"
└── AI可以基于投入数据给出进度预警
```

## 6. AI 系统设计

### 6.1 AI 能力分层

```
Level 1 - 基础能力（低API消耗，高频使用）
├── 日结算鼓励话术生成
├── 每日建议生成（基于昨日数据）
├── 未完成任务的温和提醒文案
├── 任务完成后的正反馈文案
└── 预估API消耗：每用户每天 3-5 次调用

Level 2 - 分析能力（中等API消耗，中频使用）
├── 任务智能拆解（大任务 → 子任务列表）
├── 四象限自动分类建议
├── 周报/月报文字总结生成
├── 时间分配分析与建议
├── 效率趋势分析
├── 任务预估时长建议（基于历史数据）
└── 预估API消耗：每用户每天 2-3 次调用

Level 3 - 规划能力（高API消耗，低频使用）
├── 基于历史数据的每日计划建议
├── 长线任务风险预警（"按当前进度可能延期"）
├── 里程碑拆解建议（用户输入大目标，AI建议里程碑划分）
├── 个性化效率提升方案
└── 预估API消耗：每用户每周 3-5 次调用
```

### 6.2 AI 有效性边界

```
AI 能做好的事（数据驱动，客观）：
├── ✅ 呈现数据事实："你今天在XX上花了3小时"
├── ✅ 识别行为模式："你通常上午效率最高"
├── ✅ 任务拆解："把'写论文'拆成具体步骤"
├── ✅ 时间分析："本周科研时间占比下降了10%"
└── ✅ 生成鼓励文案：基于完成情况的个性化鼓励

AI 做不好的事（需要领域知识，主观）：
├── ❌ 判断科研质量："你的论文写得好不好"
├── ❌ 判断任务难度："这篇论文应该很容易读完"
├── ❌ 替代导师指导："你应该用XX方法做研究"
├── ❌ 判断效率原因："你今天效率低是因为..."
└── ❌ 预测精确时间："你的论文还需要23天完成"

应对策略：
├── AI 只在能力边界内给建议
├── 涉及学术判断时，AI 明确说"这需要和导师讨论"
├── 所有AI建议都标注"仅供参考"
├── 用户反馈机制：每条AI建议都有 👍👎 按钮
└── 基于用户反馈持续优化prompt

```

### 6.5 AI 调用策略与成本控制

```

成本控制策略：

1. 本地缓存
   ├── 常用的鼓励话术预生成一批，存在本地
   ├── 简单场景（如"任务完成"）直接用本地话术，不调API
   ├── 只有需要个性化分析时才调用API
   └── 预估可减少 40% 的API调用

2. 批量调用
   ├── 日结算时一次性生成所有需要的文案
   ├── 而不是每完成一个任务就调一次API
   └── 减少API调用次数

3. 分级调用
   ├── Level 1 功能：优先使用本地模板 + 变量替换
   ├── Level 2 功能：使用轻量模型（如 GPT-4o-mini）
   ├── Level 3 功能：使用完整模型（如 GPT-4o / Claude）
   └── 用户可在设置中选择AI质量等级（影响API消耗）

4. API Key 管理
   ├── MVP阶段：用户自备API Key（OpenAI / Claude / 其他）
   ├── 后期：提供官方API Key（付费订阅模式）
   ├── 支持多个API提供商（OpenAI、Anthropic、本地模型）
   └── API Key 加密存储在本地

5. 离线降级
   ├── 无网络时，AI功能降级为本地预设话术
   ├── 任务拆解降级为手动拆解
   ├── 核心功能（计时、任务管理）完全离线可用
   └── 网络恢复后自动同步需要AI处理的内容

预估月度成本（单用户）：
├── 轻度使用（仅日结算话术）：约 $0.5/月
├── 中度使用（+ 任务拆解 + 周报）：约 $2/月
├── 重度使用（全部AI功能）：约 $5/月
└── 使用 GPT-4o-mini 可降低约 60% 成本

```

### 8.3 本地数据策略

```
存储位置：
├── macOS：~/Library/Application Support/FocusLab/
├── Windows：%APPDATA%/FocusLab/
├── Linux：~/.config/FocusLab/
└── 数据库文件：focuslab.db（SQLite）

备份策略：
├── 每次应用启动时，自动备份数据库到 backups/ 目录
├── 保留最近 7 个备份，自动清理更早的
├── 用户可以手动触发"导出全部数据"（JSON / CSV）
└── 后期云同步时，本地数据作为主副本

数据安全：
├── API Key 使用系统钥匙串加密存储
│   ├── macOS：Keychain
│   ├── Windows：Credential Manager
│   └── Linux：libsecret
├── 数据库本身不加密（本地应用，无远程风险）
├── 导出数据时，API Key 不包含在导出文件中
└── 删除应用时提醒用户是否清除数据

数据量预估（单用户使用 1 年）：
├── tasks：约 2000 条（日均 5-6 条）
├── sessions：约 3000 条（日均 8-10 条）
├── interruptions：约 1500 条
├── settlements：365 条
├── ai_feedbacks：约 1500 条
├── 总数据库大小：约 10-30 MB
└── SQLite 完全能胜任，无需更重的数据库
```

```
问题：浏览器 setInterval 在标签页不活跃时精度下降
解决方案：
├── 方案A：Rust 后端维护一个精确计时器
│   ├── 使用 tokio::time::interval
│   ├── 每秒通过 Tauri Event 广播到前端
│   └── 前端只负责展示，不负责计时
├── 方案B：前端使用 Web Worker
│   ├── Worker 线程不受标签页可见性影响
│   └── 兜底方案
├── 最终方案：Rust 后端计时 + 前端 Worker 双保险
│   ├── 正常情况使用 Rust 后端时间
│   ├── 前端 Worker 作为校验
│   └── 两者偏差 > 2秒时以 Rust 为准
└── 注意：倒计时结束事件由 Rust 后端触发，保证可靠性

5. AI 流式响应

实现方式：
├── Rust 使用 reqwest 发起 SSE 请求
├── 逐 chunk 通过 Tauri Event 推送到前端
├── 前端逐字显示（打字机效果）
├── 用户可以中途取消生成
└── 超时处理：30秒无响应自动降级为本地话术

示例 Rust 代码片段：
// AI 流式调用（伪代码）
async fn stream_ai_response(prompt: &str, app: AppHandle) {
    let mut stream = client.post(url)
        .json(&body)
        .send().await?.bytes_stream();

    while let Some(chunk) = stream.next().await {
        let text = parse_sse_chunk(chunk?);
        app.emit("ai:stream-chunk", &text)?;
    }
    app.emit("ai:stream-done", ())?;
}

6. 数据库迁移

简易迁移系统：
├── migrations/ 目录下按编号存放 SQL 文件
├── 数据库中维护一个 _migrations 表记录已执行的迁移
├── 应用启动时自动检查并执行未执行的迁移
├── 迁移只支持正向（不支持回滚，简化实现）
└── 示例：

CREATE TABLE IF NOT EXISTS _migrations (
    version INTEGER PRIMARY KEY,
    name    TEXT NOT NULL,
    applied_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);

-- 启动时：
-- 1. 查询 _migrations 表获取已执行的最大版本号
-- 2. 扫描 migrations/ 目录，找到版本号更大的文件
-- 3. 按顺序执行，每个成功后插入 _migrations 记录
-- 4. 任何一个失败则中止并报错
```

## 11. 风险与应对策略

### 11.1 技术风险

```
风险1：Tauri 多窗口稳定性
├── 描述：悬浮球窗口 + 主窗口 + 快捷面板的多窗口协调可能不稳定
├── 可能性：中
├── 影响：高（核心体验受损）
├── 应对：
│   ├── 早期验证：Week 1 即搭建多窗口原型
│   ├── 降级方案：如果多窗口不稳定，悬浮球改为主窗口内的固定区域
│   └── 持续关注 Tauri 2.x 的更新和社区反馈
└── 状态：需要 Week 1 验证

风险2：悬浮球跨平台一致性
├── 描述：置顶/透明/拖拽在 macOS/Windows/Linux 上行为可能不一致
├── 可能性：高
├── 影响：中
├── 应对：
│   ├── MVP 阶段优先支持 macOS（目标用户较多使用 Mac）
│   ├── Windows 作为次优先级
│   ├── Linux 最后适配
│   └── 每个平台单独测试悬浮球行为
└── 状态：已知风险，分平台处理

风险3：AI API 可用性与延迟
├── 描述：国内网络环境下 OpenAI/Anthropic API 可能不稳定
├── 可能性：高（中国用户）
├── 影响：中（AI是增强功能，不是核心功能）
├── 应对：
│   ├── 所有核心功能离线可用（计时/任务/结算）
│   ├── AI 功能有完善的离线降级方案
│   ├── 支持配置代理/自定义 API 地址
│   ├── 后期支持国产大模型 API（通义千问/文心一言等）
│   └── 支持本地模型（Ollama）
└── 状态：已在架构中考虑

风险4：SQLite 并发性能
├── 描述：多窗口同时读写 SQLite 可能导致锁冲突
├── 可能性：低（单用户应用，并发量极低）
├── 影响：低
├── 应对：
│   ├── 使用 WAL 模式提高并发读写能力
│   ├── 所有数据库操作通过 Rust 后端统一管理
│   ├── 前端不直接操作数据库
│   └── 写操作使用事务保证一致性
└── 状态：低风险，已在架构中预防

风险5：计时器精度
├── 描述：WebView 标签页不活跃时 setInterval 精度下降
├── 可能性：确定会发生
├── 影响：高（番茄钟是核心功能）
├── 应对：
│   ├── Rust 后端作为计时主源（tokio::time::interval）
│   ├── 前端 Web Worker 作为校验
│   ├── 计时结束事件由 Rust 触发
│   └── 详见 9.7 节技术方案
└── 状态：已有明确解决方案
```

```markdown
3. 功能渗透率
├── 番茄钟使用率 → 目标 ≥ 90%（活跃用户中）
├── 日结算使用率 → 目标 ≥ 70%
├── 四象限分类使用率 → 目标 ≥ 50%
├── AI 拆解使用率 → 目标 ≥ 30%
├── 长线目标使用率 → 目标 ≥ 40%
├── 周报查看率 → 目标 ≥ 25%
└── 定义：过去 7 天内使用过该功能的活跃用户比例

4. AI 质量指标
├── AI 建议采纳率 → 目标 ≥ 60%
├── AI 拆解采纳率（全部采纳 + 部分修改）→ 目标 ≥ 70%
├── AI 反馈正面率（👍 / (👍+👎)）→ 目标 ≥ 80%
└── AI 调用失败率 → 目标 < 5%

5. 健康指标（防止产品制造焦虑）
├── C级日结算占比 → 警戒线 ≤ 30%
│   └── 如果超过 30% 的日结算都是 C 级，说明用户普遍过度规划
├── 任务搁置率 → 警戒线 ≤ 25%
│   └── 如果超过 25% 的任务被搁置，说明任务拆解或引导有问题
├── 连续 C 级用户比例 → 警戒线 ≤ 10%
│   └── 连续 3 天 C 级的用户超过 10%，需要紧急优化
└── 卸载前行为分析 → 关注卸载前 3 天的使用模式
```

### 13.4 设计决策记录（ADR）

```
ADR-001：选择 Tauri 而非 Electron
├── 决策日期：2025-07
├── 背景：需要一个跨平台桌面应用框架
├── 选项：
│   ├── A. Electron — 成熟生态，但打包体积大（100MB+），内存占用高
│   ├── B. Tauri 2 — 轻量（<10MB），Rust后端性能好，但生态较新
│   └── C. Flutter Desktop — 跨平台，但桌面端不成熟
├── 决策：选择 Tauri 2
├── 理由：
│   ├── FocusLab 需要常驻后台，内存占用敏感
│   ├── Rust 后端适合处理计时器、数据库等系统级操作
│   ├── 打包体积小，对用户友好
│   ├── Tauri 2.x 已足够稳定
│   └── 团队有 Rust 学习意愿
└── 风险：Tauri 生态不如 Electron 成熟，遇到问题可能需要自己解决

ADR-002：选择本地 SQLite 而非云数据库
├── 决策日期：2025-07
├── 背景：需要存储用户的任务、计时、设置等数据
├── 选项：
│   ├── A. SQLite 本地存储
│   ├── B. Supabase 云端
│   └── C. IndexedDB（浏览器存储）
├── 决策：选择 SQLite
├── 理由：
│   ├── 隐私优先：用户数据不离开本地
│   ├── 零服务器成本：MVP 阶段不需要后端
│   ├── 离线可用：研究生可能在无网环境工作
│   ├── SQLite 性能对单用户应用绰绰有余
│   └── 后期可以增加云同步层
└── 风险：多设备同步需要额外开发

ADR-003：AI 功能用户自备 API Key
├── 决策日期：2025-07
├── 背景：AI 功能需要调用大模型 API，产生费用
├── 选项：
│   ├── A. 用户自备 API Key
│   ├── B. 开发者提供 API Key，按使用量收费
│   └── C. 完全不做 AI 功能
├── 决策：MVP 阶段选择 A，后期考虑 B
├── 理由：
│   ├── 零运营成本：开发者不需要垫付 API 费用
│   ├── 用户控制：用户选择自己信任的 AI 提供商
│   ├── 隐私：API 调用直接从用户设备发起
│   ├── 降级方案：无 API Key 也能使用所有核心功能
│   └── 研究生群体获取 API Key 的门槛不高
└── 风险：可能劝退非技术用户，后期需要提供更简单的方案

ADR-004：不使用第三方 UI 组件库
├── 决策日期：2025-07
├── 背景：需要统一且有品牌特色的 UI 组件
├── 选项：
│   ├── A. 使用 Element Plus / Naive UI 等成熟组件库
│   ├── B. 使用 Headless UI（如 Radix Vue）+ 自定义样式
│   └── C. 完全自建组件
├── 决策：选择 C（核心组件自建），B 作为辅助
├── 理由：
│   ├── 产品视觉风格独特（呼吸灯、粒子特效、彩色进度条）
│   ├── 第三方组件库的设计语言不匹配
│   ├── 组件数量不多（约 30-40 个），自建可控
│   ├── TailwindCSS 加速样式开发
│   └── 复杂交互组件（如 Modal、Tooltip）可借助 Headless UI
└── 风险：开发速度较慢，需要更多时间处理边界情况

ADR-005：日结算 C 级不使用红色
├── 决策日期：2025-07
├── 背景：日结算需要为完成率低于 70% 的用户提供视觉反馈
├── 选项：
│   ├── A. 红色进度条 + 警告文案
│   ├── B. 灰色进度条 + 温和文案
│   └── C. 不做视觉区分
├── 决策：选择 B
├── 理由：
│   ├── 产品原则"陪伴不监工"
│   ├── 红色会触发负面情绪联想（失败、错误、危险）
│   ├── 研究生群体本身焦虑感强，不应加重
│   ├── 灰色传达"今天比较平淡"而非"你失败了"
│   ├── 温和文案提供心理安全感
│   └── 用户反馈可验证：如果用户因 C 级而改善行为 → 有效
│       如果用户因 C 级而逃避使用 → 需要调整
└── 风险：可能有用户觉得反馈不够强烈，通过语气设置中的"极简数据型"满足

ADR-006：计时器主逻辑放在 Rust 后端
├── 决策日期：2025-07
├── 背景：番茄钟计时的精度需要保障
├── 选项：
│   ├── A. 前端 setInterval
│   ├── B. 前端 Web Worker
│   ├── C. Rust 后端 tokio::time
│   └── D. C + B 双保险
├── 决策：选择 D
├── 理由：
│   ├── 前端 setInterval 在标签页不活跃时精度极差
│   ├── Rust 后端计时不受 WebView 状态影响
│   ├── 前端 Worker 作为展示层校验
│   ├── 计时结束的关键事件由 Rust 触发，保证可靠性
│   └── 即使前端崩溃，后端仍能记录计时数据
└── 风险：增加了架构复杂度，但计时是核心功能，值得投入
```
