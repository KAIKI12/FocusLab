## 2. 专注计时模块

### 2.1 番茄钟模式

基于番茄工作法的计时系统,支持多种时长模式,与任务系统深度集成。

**计时模式**:
- 🍅 经典模式: 25分钟专注 + 5分钟休息
- 🍅🍅 深度模式: 45分钟专注 + 10分钟休息
- 🍅🍅🍅 沉浸模式: 90分钟专注 + 15分钟休息
- ⏱️ 自由模式: 不设时限,手动开始/结束

**长休息规则**: 每完成 4 个番茄钟,触发一次长休息(15-30分钟)

**计时状态机**:
```
┌──────┐    开始    ┌──────┐   时间到   ┌──────┐
│ 空闲  │ ────────→ │ 专注中 │ ────────→ │ 休息中 │
└──────┘           └──────┘           └──────┘
                     │  ↑                │
                暂停  │  │ 继续           │ 休息结束
                     ↓  │                ↓
                   ┌──────┐          ┌──────┐
                   │ 暂停中 │          │ 空闲  │
                   └──────┘          └──────┘
                     │
                放弃  │
                     ↓
                   ┌──────┐
                   │ 已放弃 │ → 记录放弃原因（可选）
                   └──────┘
```

**自由模式详细规则（v1.2.3 补充)**:

研究生写论文、读长文献、深度思考等场景不适合 25 分钟切块,自由模式作为"秒表式"计时提供无时限保底。

| 维度 | 番茄模式 🍅 | 自由模式 🌀 |
|------|-----------|-----------|
| 时长 | 固定(25/45/90) | 不设上限 · 手动结束 |
| 数字显示 | 倒计时(递减) | 正计时(递增) |
| 视觉 | 实心环,随时间填充消减 | 虚线环慢速旋转示意"计时在跑" · 紫罗兰 `#8B5CF6` |
| 主色 | Primary Blue | Violet `#8B5CF6` / `#A78BFA` |
| 自动休息 | ✅ 时间到自动进入休息 | ❌ 不自动 · 手动结束后回空闲 |
| 休息三选一弹窗 | ✅ | ❌ |
| 计入 `pomodoro_count` | ✅ | ❌ (只计入 `focus_duration_sec`) |
| 中断记录 | 正常记录 | 正常记录(暂停/切换任务会打断,与番茄一致) |
| `sessions.mode` 值 | `pomodoro` | `free` |
| 最长单次会话 | 固定上限 | 软上限 4h · 超过提示"要不要休息一下?" |
| 典型场景 | coding · 做题 · 事务性任务 · 有明确时间预算的工作 | 写作 · 读论文 · 深度思考 · 有创造性、不适合打断的工作 |

**切换方式**:
- **启动前**:焦点卡 meta 行有模式切换 chip(🍅 番茄 / 🌀 自由)· 单击切换
- **命令面板**:输入"切换计时模式",或直接输入"自由" / "番茄"匹配
- **计时中**:**不允许切换**(结束当前会话才能换)· UI 上 chip 变灰不可点
- **默认**:首次启动为 🍅 番茄 · 用户上次用过的模式作为下次默认

**暂停/继续交互**:
- 主窗口焦点卡:主按钮 ⏸/▶
- 悬浮球:**双击**触发(专注中/自由计时中均支持,v1.2.3)
- 快捷键:`Space`
- 命令面板:"暂停 / 继续专注"

**自由模式偏好(v1.2.3 · `设置 → 番茄钟 → 🌀 自由计时模式` 分区)**:

| 偏好项 | 类型 | 默认 | 行为 |
|--------|------|------|------|
| 默认计时模式 | 🍅 番茄 / 🌀 自由 / 上次使用 | 🍅 番茄 | 按下主按钮、从任务启动专注时使用的模式 |
| 最小记录时长 | 1–15 分钟 · 滑杆 | 3 分钟 | 自由模式短于此时长的会话丢弃,不计入 `focus_duration_sec`、不出现在周报。避免误触污染数据 |
| 超时柔提醒 | 30–180 分钟 · 滑杆 + 开关 | 90 分钟 · 开 | 连续自由计时超过此时长时,悬浮球轻微呼吸提示。**不强制打断、不弹窗**,与 ADR-005"不监工"一致 |
| 暂停计为中断 | 开关 | 关 | 开启时按暂停直接结束会话并写入 `interruptions`;关闭时仅暂停,可恢复(与番茄模式一致) |
| 切换任务自动结束当前会话 | 开关 | 开 | 保证每段专注归属唯一的任务,便于归档与周报 |

### 2.2 中断与切换

**中断处理**:
- 用户点击"暂停"时,弹出中断原因选择:
  - 🧑‍🏫 导师/同学找
  - 📞 电话/消息
  - 🚶 主动休息
  - 🔄 切换任务
  - 📝 其他
- 中断数据被记录,用于分析时间碎片化程度
- 中断原因选择是可选的,用户可以直接暂停不填

**任务切换**:
- 在专注中点击"切换任务"
- 当前任务自动暂停并记录已用时间
- 弹出任务列表,选择新任务
- 新任务开始计时
- 切换记录保存,用于分析多任务切换频率

**回溯补录**:
- 用户忘记按"开始"时,可以事后补录
- 选择任务 → 输入开始时间和结束时间
- 补录的时间段标记为"手动补录"
- 补录数据参与统计,但在分析中标注来源

### 2.4 休息结束交互增强（v1.1 新增）

番茄钟休息结束时,提供三个选项而非仅"继续/延长":

```
┌─────────────────────────────────────────────────┐
│ ☕ 休息结束！                                    │
│                                                 │
│ 上一个番茄钟：复现baseline代码（25分钟）         │
│ 今日已完成：🍅🍅🍅⚪⚪⚪ 3/6 个番茄钟            │
│                                                 │
│ ┌─────────────────────────────────────────────┐ │
│ │ [▶ 继续：复现baseline代码]                   │ │
│ │     继续刚才的任务，再来一个番茄钟            │ │
│ ├─────────────────────────────────────────────┤ │
│ │ [🔄 切换到其他任务]                          │ │
│ │     选择另一个任务开始新的番茄钟              │ │
│ ├─────────────────────────────────────────────┤ │
│ │ [☕ 再休息 5 分钟]                           │ │
│ └─────────────────────────────────────────────┘ │
│                                                 │
│ 💡 AI建议：接下来处理"回复导师邮件"（紧急重要）│
└─────────────────────────────────────────────────┘
```

**AI 建议逻辑**:
- 优先推荐第一象限（紧急重要）的未完成任务
- 如果第一象限已清空,推荐第二象限中关联长线目标的任务
- 如果用户已连续3个番茄钟做同一任务,建议切换（避免疲劳）
- 考虑用户能量状态: 高能量推荐核心科研,低能量推荐事务性任务
- AI 建议只是提示文字,不阻塞用户操作

### 2.3 计时数据记录

每个专注时段记录以下数据:

```json
{
  "session_id": "唯一标识",
  "task_id": "关联的任务",
  "start_time": "开始时间",
  "end_time": "结束时间",
  "planned_duration": "计划时长（番茄钟模式下）",
  "actual_duration": "实际时长",
  "mode": "pomodoro | free",
  "status": "completed | paused | abandoned",
  "interruptions": [
    { "time": "中断时间", "reason": "中断原因", "duration": "中断时长" }
  ],
  "is_manual_entry": "是否手动补录"
}
```

## 3. 任务管理模块

### 3.1 任务创建

**任务基本属性**:
- 任务名称（必填）
- 任务描述（可选）
- 所属分类（科研核心/事务性/学习提升/日常例行/个人发展）
- 四象限位置（紧急重要/不紧急重要/紧急不重要/不紧急不重要）
- 预估时长（可选,AI可辅助预估）
- 关联长线目标（可选）
- 关联里程碑（可选）
- 截止日期（可选,即"到期日 `due_date`" · 见 §3.7）
- 重复规则（可选: 每天/每周/自定义）
- 标签（自定义标签,如"论文A""项目B"）

**任务创建方式**:
1. 手动创建（填写表单）
2. 快速创建（只输入名称,其他后补）
3. AI拆解创建（输入大任务,AI拆解为子任务）
4. 从昨日遗留中继承
5. 从引导思考流程中生成

### 3.2 四象限分类系统

```
              紧急                    不紧急
        ┌──────────────────┬──────────────────┐
        │                  │                  │
  重要  │  🔴 第一象限      │  🟡 第二象限      │
        │  立即执行         │  计划执行         │
        │                  │                  │
        │  例：明天DDL的报告│  例：论文写作      │
        │  导师今天要的数据 │  学习新方法       │
        │                  │                  │
        ├──────────────────┼──────────────────┤
        │                  │                  │
 不重要 │  🟠 第三象限      │  🟢 第四象限      │
        │  委托或快速处理   │  减少或搁置       │
        │                  │                  │
        │  例：回复非紧急邮件│  例：整理桌面     │
        │  填写报销单       │  刷学术八卦       │
        │                  │                  │
        └──────────────────┴──────────────────┘
```

**AI 辅助分类**:
- 用户创建任务时,AI 根据任务描述建议象限位置
- 用户可以接受或手动调整
- AI 学习用户的分类习惯,建议越来越准确
- 每日计划时,AI 建议优先处理第一象限,然后第二象限

### 3.3 任务状态流转

```
任务生命周期：

  ┌──────┐   开始    ┌──────┐   完成    ┌──────┐
  │ 待开始 │ ───────→ │ 进行中 │ ───────→ │ 已完成 │
  └──────┘          └──────┘          └──────┘
     │                 │
     │ 搁置            │ 搁置
     ↓                 ↓
  ┌──────┐          ┌──────┐
  │ 已搁置 │ ←────── │ 已搁置 │
  └──────┘          └──────┘
     │
     │ 恢复
     ↓
  ┌──────┐
  │ 待开始 │ （重新进入队列）
  └──────┘
```

**搁置规则**:
- 用户可以主动搁置任何任务
- 连续3天未处理的遗留任务,系统建议搁置
- 搁置时可选填原因（任务太难/优先级变了/不想做了）
- 搁置区的任务不参与日结算统计
- 搁置超过30天的任务,提醒用户是否永久删除
- 搁置区位于界面底部折叠区域,不显眼

### 3.4 任务分类体系

**按性质分类（一级分类）**:
- 📖 科研核心: 读论文、写论文、做实验、跑代码、数据分析
- 📋 事务性: 报销、填表、回邮件、组会PPT、行政事务
- 📚 学习提升: 上课、学新工具、读教材、参加讲座
- 🔄 日常例行: 锻炼、吃饭、通勤、固定会议
- 🌟 个人发展: 社交、实习准备、副业、兴趣爱好

**按紧急程度叠加（二级标签）**:
- 🔴 今日必须完成（DDL驱动）
- 🟡 本周内完成（重要但不紧急）
- 🟢 持续推进（长线任务的日常投入）

两个维度正交组合,形成完整的任务画像。

### 3.5 任务排序与优先级调整（v1.1 新增）

**同一象限内的默认排序规则**（叠加优先级,从高到低）:
1. 手动置顶的任务排最前
2. 有截止日期的按截止日期升序（越近越前）
3. 关联长线目标的排在未关联的前面
4. 来源为 carried_over 的排在前面（昨天没完成的优先处理）
5. 预估时长短的排在前面（鼓励快速清理小任务）
6. 按 sort_order 字段（手动拖拽调整的顺序）
7. 按 added_at 时间（先添加的在前）

**排序交互**:
- 拖拽排序: 长按/拖拽手柄调整顺序,跨象限拖拽同时修改排序和四象限分类
- 快捷操作（右键菜单）: 📌 置顶 / ⬆️ 上移 / ⬇️ 下移 / 🔄 移到其他象限
- AI 排序建议（Phase 2）: 基于截止日期、目标关联度、能量状态等因素给出排序建议

**用户可在设置中选择排序策略**: 默认智能排序 / 手动排序 / 按截止日期 / 按创建时间 / 按预估时长

### 3.6 多任务并行 — 后台任务标记（v1.1 新增）

研究生经常"同时推进多件事"（如跑代码的同时看论文）。设计一个轻量的后台任务机制:

**设计思路**: 不做复杂的多任务同时计时,而是允许标记某些任务为"后台进行中"。同一时间仍然只有一个"主任务"在计时。

**tasks 表新增字段**: `is_background BOOLEAN DEFAULT 0`

**后台任务的行为**:
- 不参与番茄钟计时,不记录 session,不纳入专注时间统计
- 但纳入完成率统计（完成了就算完成）
- 在任务列表中显示 ⏳ 图标和"后台"标签
- 日结算中单独展示: "后台完成的任务: 2个"

**交互设计**:
- 创建任务时多一个开关: "这是一个后台/低注意力任务"
- 或在已有任务上右键 → "标记为后台任务"
- 后台任务在列表中排列在所有主动任务之后
- 不显示 ▶ 开始按钮,只显示 ✓ 完成按钮

**典型场景**: 代码在跑/模型在训练、等导师回复、下载大文件

### 3.7 任务到期日与日历待办（v1.2.2 新增）

**设计动机**: 研究生常遇到"某任务必须在某天前交"的场景(开题报告、基金申请、会议 DDL 等),需要"记一笔 + 临近提醒 + 到期自动浮到眼前"的轻量能力。不做外部系统日历集成(见 `docs/06 ADR-012`),只用内部日历 + 本地通知完成闭环。

#### 字段语义(tasks.due_date)

| 概念 | 含义 | 区别于 |
|------|------|--------|
| **到期日 `due_date`** | 任务**必须完成的最晚日期**(可选,DATE 类型,无时分) | ≠ 计划执行日(任务可在 due_date 之前任意一天动手) |
| **计划日 `plan_date`**(由 daily_task_assignments 表隐式承载) | 用户把任务放到"哪一天的今日计划"里 | ≠ 到期日(可远早于 due_date) |
| **完成时间 `completed_at`** | 实际标记完成的 TIMESTAMP | ≠ 到期日 |

**语义示例**: 任务"交开题报告",`due_date = 2026-05-10`(5/10 前必须交),用户可以在 4/25 就完成它,也可以拖到 5/10 当天。若 5/11 仍未完成则 `is_overdue = true`。

#### 四种视觉状态

延续 ADR-005 "陪伴不监工"精神,克制用红色,以暖橙/琥珀为主色:

| 状态 | 触发条件 | chip 样式 | 示例文案 |
|------|---------|-----------|---------|
| 普通 | `due_date` 在 3 天之后 | `.chip-due`(灰) | `📅 5/10` |
| soon | 3 天内(不含当日) | `.chip-due.is-soon`(琥珀) | `📅 4/21 · 3 天后` |
| today | `due_date = 今日逻辑日` | `.chip-due.is-today`(暖橙 + 加粗) | `⏰ 今日到期` |
| overdue | `due_date < 今日` 且未完成 | `.chip-due.is-overdue`(深橙 + pulse 动画) | `⚠ 逾期 2 天` |

#### 产品闭环

```
用户给任务设 due_date  
        ↓  
  [前一天 20:00 · 可自定义时间]  
        ↓  
  系统通知 + 悬浮球红点:"⏰ XX 任务明天到期"  
        ↓  
  [到期当日 · 用户首次打开 / 逻辑日切换时]  
        ↓  
  自动置顶到今日计划(不改象限分类,只加 chip-due.is-today)  
  主屏顶部"⏰ N 个今日到期 · M 个逾期"总览条  
        ↓  
  完成 → 正常结算  
  未完成 → 次日自动继承为"逾期",打 chip-due.is-overdue  
```

#### 内部日历视图联动(`calendar.html`)

- 月视图格子展示当日 DDL 任务: `event-pill.due`(普通)/ `event-pill.due.is-today`(今日)
- 右栏 `day-detail` 在"今日安排"之上插入"⏰ 今日到期"小节,高亮显示
- 月视图图例新增"任务到期日"项

#### 任务列表联动(`main-tasks.html`)

- 列表视图: 任务 meta 行加 `chip-due`
- 四象限视图: qt-card 的 qt-meta 加 `chip-due`
- 排序策略延续 §3.5: **有截止日期的按截止日期升序**,DDL 临近自然靠前

#### 主屏联动(`main-today.html`)

- 任务面板顶部"⏰ 到期总览"提示条
- 各象限内任务不改分类,只附 `chip-due` 表意
- **自动置顶逻辑**: `due_date = 今日` 且未完成 → 由系统自动创建当日 `daily_task_assignments` 记录,用户打开 FocusLab 时看到这些任务已在今日计划里

#### 明确不做的事

- ❌ 不做外部日历同步(见 ADR-012)
- ❌ 不做 ICS 导出/订阅
- ❌ 不做到期"强制弹窗"打断用户专注(只用悬浮球红点 + 通知)
- ❌ 不做"逾期"的红色/批判性文案(延续 ADR-005)

### 4.2 引导思考流程

**设计原则**:
- 不是"问答式访谈",而是"启发式卡片"
- 默认不强制,用户可以跳过
- 根据用户行为自适应展示
- 目标: 3分钟内完成

**触发策略**:
- 新用户（前7天）: 默认展示引导流程
- 老用户:
  - 如果用户能快速添加任务 → 不展示引导
  - 如果用户打开应用后30秒无操作 → 主动弹出
  - 用户可以在设置中选择"总是展示/从不展示/智能判断"
- 引导流程可以在任何时候通过快捷键唤起

**引导流程（5步,每步可跳过）**:

**Step 1 - 回顾昨日**: 展示昨日完成情况,让用户选择未完成任务是否继续

**Step 2 - 审视长线目标**: 展示长线目标进度,让用户选择今天推进哪个方向

**Step 3 - 检查固定日程**: 展示今日固定日程,计算可用工作时间

**Step 4 - 设定今日目标**: AI基于长线目标和昨日数据建议今日任务,用户可采纳/调整/自写

**Step 5 - 状态检查**: 选择今日能量水平（低能量/一般/不错/充沛）,AI据此调整建议

## 5. 日结算模块

### 5.1 结算触发机制

**方式A: 手动触发（推荐）**
- 用户点击"结束今天"按钮

**方式B: 智能触发**
- 检测到用户结束最后一个任务后 10 分钟无操作
- 弹出提示: "今天的工作结束了吗？要进行日结算吗？"

**方式C: 定时触发（兜底）**
- 用户在设置中自定义结算时间（默认 22:00）
- 到达时间后弹出提醒
- 如果用户忽略,不再重复弹出（第二天早上再展示）

**优先级**: 方式A > 方式B > 方式C

### 5.2 结算分级系统

| 等级 | 条件 | 视觉效果 | 音效 | AI文案风格 |
|------|------|----------|------|-----------|
| S级 | 完成率 ≥ 120%（超额完成） | 金色光效 + 彩虹渐变进度条 + 粒子特效 | 成就解锁音效 | 热烈祝贺 + 数据亮点 |
| A级 | 完成率 = 100%（完美完成） | 发光进度条 + 星星特效 | 清脆的完成音 | 肯定 + 明日展望 |
| B级 | 完成率 70%-99%（基本完成） | 蓝色进度条,柔和 | 柔和提示音 | 肯定已完成的 + 温和提及未完成 |
| C级 | 完成率 < 70%（需要调整） | 灰蓝色进度条（不用红色） | 无音效或极轻柔提示 | 理解 + 建议调整 + 鼓励明天重新开始 |

**完成率计算**（v1.1 修复）:

基于 `daily_task_assignments` 表,引入"计划锁定"机制:

**计划锁定时机**（按优先级,先满足的生效）:
1. 用户完成引导思考流程 → 自动锁定
2. 用户点击"确认今日计划" → 手动锁定
3. 用户开始第一个番茄钟 → 自动锁定
4. 当天第一次添加任务后15分钟 → 自动锁定

锁定前的任务为"计划内任务"（`is_planned=true`）,锁定后新添加的为"额外任务"（`is_planned=false`）。

**计算公式**:
- 计划任务数 = COUNT(is_planned=true AND day_status!='cancelled')
- 计划完成数 = COUNT(is_planned=true AND day_status='completed')
- 额外完成数 = COUNT(is_planned=false AND day_status='completed')
- 基础完成率 = 计划完成数 / 计划任务数 × 100%

**评级规则**:
- S级: 基础完成率=100% 且 额外完成数≥1（全部完成+还有余力）
- A级: 基础完成率=100% 且 额外完成数=0（恰好完成所有计划）
- B级: 基础完成率 ≥70% 且 <100%
- C级: 基础完成率 <70%
- 注意: 评级只基于计划内任务,额外任务在结算面板中独立展示,避免"多添加多完成冲S级"

### 5.4 未完成任务处理

**温和处理机制**:

```
┌─────────────────────────────────────────┐
│ 📋 这些任务今天没有完成                  │
│                                         │

│   [明天继续] [搁置] [已经差不多了✓]     │

│    重要的是你今天确实在推进。            │
└─────────────────────────────────────────┘
```

**自动降级规则**:
- 第1天未完成: 正常显示在"昨日遗留"
- 第2天仍未完成: 标记为"连续遗留",AI建议拆分或调整
- 第3天仍未完成: 系统建议搁置
- 用户选择搁置后: 任务进入搁置区,不再主动提醒
- 搁置30天后: 提醒用户是否永久删除

**关键原则**:
- 永远不用"失败""落后""拖延"等词汇
- 用"调整""搁置""重新安排"替代
- 给用户"体面退出"的选项
- 搁置不是失败,是策略性选择

### 5.5 跨天工作边界处理规则（v1.1 新增）

研究生常熬夜工作,需要明确跨天场景的处理规则。

**逻辑日概念**: 一个"逻辑日"不等于自然日（0:00-23:59）,而是从日期分界线到下一个分界线。默认分界线为凌晨4:00,用户可自定义。

**Session 归属规则**: 按 start_time 的逻辑日归属,不拆分。
- 23:30开始、00:05结束 → 归属当天（start_time ≥ 4:00）
- 03:30开始、04:20结束 → 归属前一天（start_time < 4:00）

**任务完成归属**: 按 completed_at 的逻辑日归属。如果任务分配在逻辑日A但在逻辑日B才完成,A的记录标记为 carried_forward,B自动创建新记录。

**日结算数据冻结**:
1. 用户手动触发日结算 → 立即冻结当天数据
2. 到达日期分界线时自动冻结 → 生成"兜底结算"（统计数字自动算,AI寄语留空）
3. settlement 记录一旦生成,统计数字不再变化（ai_summary 和 user_reflection 可后续补写）

## 5.6 主观信号体系（v1.2 新增 · 🧪 实验功能）

> **v1.2 定位**: 本节涉及的"晨间意图档位"与"晚间真实情绪"为**实验功能,默认关闭**。  
> 启用路径: 设置 → 实验功能 → 早/晚心情打卡。  
> 理由: 避免新用户认知过载(见 `docs/06` ADR-010 主屏密度策略),先让核心番茄+任务+结算闭环跑通,再按数据决定是否默认开启。  
> 里程碑主观进度感受沿用既有设计,不是实验功能。

### 设计动机

客观数据（完成率/专注时长/任务数）只能反映**做了什么**,无法反映**感觉怎样**。
用户可能 100% 完成任务但精疲力尽,也可能只完成 50% 却心态积极。
为了完整记录一天,FocusLab 收集三类**主观信号**,与客观数据分开存储、**不参与 S/A/B/C 评级计算**。

### 三类信号统一模型

| 信号 | 粒度 | 语义 | 触发场景 | 输入形式 |
|------|------|------|----------|----------|
| 晨间意图档位 🧪 | 日 | **承诺** · "今天想成为几档能量" | 打开 main-today 首次/每日首开 · 仅在实验功能启用时 | 五档 emoji(🌙🌤☀️⚡🔥) |
| 晚间真实情绪 🧪 | 日 | **反馈** · "今天真实感受" | 日结算 Step 2(摘要后、评级前) · 仅在实验功能启用时 | 五档 emoji(😔😕😐🙂😍) |
| 里程碑主观进度感受 | 里程碑 | **评估** · "感觉推进到哪了" | 长线目标面板手动标注 | 刚开始 ↔ 快完成了 滑块 |

三者共用数据字段 `kind / scale / value / note / captured_at`,区别仅在 `kind`。

### 核心原则

**P1 · 解耦原则**: 主观信号不改变任务引擎行为 —— 不调任务难度、不调番茄时长、不改 S/A/B/C 评级、不触发 AI 寄语语气变化。

> 理由: 如果"评低分 → 系统降低任务难度",会形成"情绪差→做更少→完成感更差→情绪更差"的负反馈循环,违反 ADR-005"陪伴不监工"精神。

**P2 · 定调原则**: 晨间意图五档**都是正向文案**,不存在"差"的档位。1 档是"保养日 · 维持状态也是一种推进",5 档是"冲刺档 · 值得被标记的一天"。

**P3 · 可跳过原则**: 三类信号均**可跳过**,跳过不影响任何其他功能(评级/统计/AI)。跳过不记为"C 级信号",只是字段为空。

**P4 · 不做好坏判断**: 早/晚对照**并列展示**"意图 / 实际",不用 `→ 提升 / 持平 / 下降` 之类的语气词,把解读权交给用户。

### 日结算三段式流程(定稿)

```
日结算面板
   │
   ├─ STEP 1 · 今日摘要       ← 纯数字,不分级
   │    • 已完成 7/8 任务
   │    • 专注 4h 12m
   │    • 番茄 6 次
   │
   ├─ STEP 2 · 真实情绪       ← 晚间主观信号打卡
   │    • 😔 😕 😐 🙂 😍     五档 emoji
   │    • 显示早/晚并列条     "今晨意图 / 今夜真实"
   │
   └─ STEP 3 · 客观评级       ← 基于完成率的 S/A/B/C
        • A 级 · 87.5% 完成率
        • AI 寄语(可选,点击生成)
```

**为什么是这个顺序**: 先展示事实数据,再问感受 —— 避免用户在信息真空里评价一天。
先自评情绪、再看评级 —— 避免被 S/A/B/C 锚定真实感受。

### 晨间意图档位的 UI 位置

- **默认状态**: 🧪 实验功能 · **默认关闭** · 用户需在 `设置 → 实验功能` 中手动开启
- **启用后位置**: 不占用 main-today 核心 4 件位置(见 ADR-010),展开在"今日切片"折叠卡内 · 当日首次打开时展开一次,用完自动收起

- **文案**: 五档全正向定调(见原型 `screens/mood-check.html` §02)

## 6. 长线目标模块

### 6.1 长线目标结构

**三层结构**:

1. **长线目标（Goal）** — 例: 发表一篇SCI论文
   - 属性: 名称、描述、预计完成时间、状态
   - 一个用户可以同时有多个长线目标

2. **里程碑（Milestone）** — 例: 完成文献综述、完成方法论设计
   - 属性: 名称、描述、预计完成时间、状态、排序
   - 每个长线目标包含多个里程碑
   - 里程碑之间有先后顺序

3. **子任务（Sub-task）** — 例: 阅读论文A、复现baseline代码
   - 属性: 与普通任务相同
   - 每个里程碑包含多个子任务
   - 子任务可以被拉入每日计划

**映射关系**: 长线目标 1:N 里程碑 1:N 子任务 → 每日任务

### 6.2 里程碑进度可视化

```
┌─────────────────────────────────────────────────┐
│ 🎯 长线目标：发表SCI论文                         │
│ 创建于 45 天前 · 已投入 32 天                    │
├─────────────────────────────────────────────────┤
│                                                 │
│ 里程碑进度：                                    │
│ [████████████████████████████░░░░░░░░░░░░░░░░]  │
│  里程碑 3/7 进行中                              │
│                                                 │
│ ✅ 里程碑1：确定研究方向          已完成（7天）  │
│ ✅ 里程碑2：完成文献综述          已完成（12天） │
│ 🔄 里程碑3：方法论设计            进行中（13天） │
│    ├── ✅ 学习XX方法              已完成         │
│    ├── 🔄 复现baseline            进行中（今日） │
│    └── ⬜ 设计改进方案            待开始         │
│ ⬜ 里程碑4：实验与数据收集        待开始         │
│ ⬜ 里程碑5：撰写初稿              待开始         │
│ ⬜ 里程碑6：导师审稿+修改         待开始         │
│ ⬜ 里程碑7：投稿+审稿回复         待开始         │
│                                                 │
│ 💭 主观进度感受：                                │
│ [刚开始] ──────────●──── [快完成了]              │
└─────────────────────────────────────────────────┘
```

**设计要点**:
- 不显示百分比,只显示里程碑完成数量
- 不预测"还要多久",只显示"已投入多少天"
- 用户可以手动标注主观进度感受
- 里程碑可以随时增删改（计划会变是正常的）
- 每个里程碑展开后显示子任务完成情况

### 6.3 长线目标与每日任务的关联

**关联方式**:

1. **创建任务时关联**: 选择"关联到长线目标" → 选择目标 → 选择里程碑
2. **引导思考时关联**: Step 2展示长线目标进度,选择推进方向后自动展示待完成子任务
3. **AI建议时关联**: AI分析进度后建议应推进的子任务,采纳后自动关联

**数据价值**:
- 用户可以看到"今天做的事对长期目标的贡献"
- 周报中展示"本周在各长线目标上的时间投入"
- AI可以基于投入数据给出进度预警

### 7.2 周报/月报

**周报内容**:
- 📊 本周概览（总专注时长、完成任务数、完成率、番茄钟数）
- 📈 与上周对比（专注时长变化、完成率变化、中断次数变化）
- 🎯 长线目标进展（各目标本周投入时间、里程碑推进情况）
- 📊 时间分布（各类别占比及变化趋势）
- 🤖 AI 周度总结（基于数据生成文字总结和建议）

**周报生成方式**:
- 自动生成: 每周日晚上自动生成并通知用户
- 手动生成: 用户随时可以点击"生成周报"
- AI参与: AI基于数据生成文字总结和建议
- 导出格式: 支持导出为 Markdown / PDF

### 7.3 数据导出详细设计（v1.1 新增）

**场景1: 周报/月报导出**
- 格式: Markdown / PDF
- 内容: 与周报展示页面一致
- 用途: 发给导师/记录科研日志

**场景2: 原始数据导出（备份/迁移）**
- 格式: JSON（结构化）/ CSV（表格工具打开）
- 内容范围用户可选: 任务数据、专注时段、长线目标、日结算记录、每日计划、AI交互记录
- 文件结构: `focuslab_export_YYYYMMDD/` 下按表分文件 + `metadata.json`
- 导出时排除 API Key

**场景3: 数据导入（从备份恢复）**
- 选择导出的 JSON 文件夹 → 预览数据条数 → 选择"覆盖"或"合并"
- MVP 阶段只做"覆盖导入",合并逻辑后期再做

**场景4: 集成导出（Phase 4+）**
- 导出为 iCal 格式、Obsidian 笔记格式、API 接口

## 8. 科研人格图鉴系统（v1.3 规划 · 🧪 实验功能）

> **定位**: 社交分享优先、数据洞察为底。核心公式: 真实行为数据 + 动物意象 + SBTI 式自嘲文案 = 又扎心又想转发。
> **灵感来源**: SBTI（Silly Big Personality Test, 2026.4 爆火）27 型人格测试的传播机制。
> **关键约束**: 所有标签描述"风格"而非"问题"（ADR-005 延伸）· 不联动任务引擎（ADR-009）· 不做排行榜 · 详见 ADR-011。

### 8.1 设计原则

1. **社交货币优先**: 人格标签的第一目标是"用户想截图分享",第二目标才是"帮用户了解自己"
2. **数据撑腰**: 每个标签都有明确的数据触发条件,不是随机分配
3. **正向/自嘲/不审判**: 所有文案允许自嘲,但不允许系统审判用户（"你是拖延症"❌ → "DDL 是你唯一的生产力来源,这一点你心知肚明"✅）
4. **动态漂移**: 人格每周更新,反映行为变化,不固定标签
5. **可跳过**: 人格系统完全可选,关闭不影响任何核心功能

### 8.3 五维匹配模型

人格匹配基于 5 个行为维度,每个维度 1-5 分:

| 维度 | 数据来源 | 低分描述 | 高分描述 |
|------|----------|----------|----------|
| 时段偏好 | 专注时段分布热力图 | 晨型(6-12点) | 夜型(20-4点) |
| DDL 模式 | 任务完成时间 vs DDL 距离 | 提前完成 | 临阵爆发 |
| 稳定性 | 日完成率标准差 | 波动型 | 稳态型 |
| 强度 | 日均专注时长 | 低频(<2h) | 高产(>6h) |
| 多线程 | 并行目标/任务数 | 单点突破 | 多线并进 |

每周基于本周数据重新计算五维得分,匹配距离最近的人格。

## 10. 桌面集成模块

### 10.1 显示模式

**模式1: 迷你悬浮球（默认工作模式）**
- 位置: 屏幕边缘,可拖拽
- 大小: 约 180×40 px
- 内容: 当前任务名 + 番茄钟倒计时
- 透明度: 70%（可调节）
- 交互: 单击展开快捷面板 / 双击打开主窗口 / 右键菜单 / 拖拽移动
- 吸附: 靠近屏幕边缘时自动吸附
- 置顶: 始终在其他窗口之上

**模式2: 侧边栏模式**
- 位置: 屏幕右侧
- 宽度: 220px,半透明
- 内容: 今日任务列表 + 进度 + 当前计时
- 交互: 可折叠/展开

**模式3: 完整窗口模式**
- 用于: 每日计划、日结算、数据分析、设置
- 大小: 可调节,默认 1000×700 px
- 触发: 双击悬浮球 / 全局快捷键 / 系统托盘
- 背景: 支持毛玻璃效果

**模式切换快捷键**:
- Cmd/Ctrl + Shift + F → 切换悬浮球/侧边栏

## 11. 首次启动体验 — FTUE（v1.1 新增）

### 11.1 设计目标

- 2分钟内完成基础设置
- 让用户立刻能创建第一个任务并开始计时
- 不要让用户面对空白界面
- 不强制完成所有设置步骤
- 传递产品核心理念: "陪伴,不监工"

### 11.2 完整流程（5步,每步可跳过）

```
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│ Step 1   │ → │ Step 2   │ → │ Step 3   │ → │ Step 4   │ → │ Step 5   │
│ 欢迎     │   │ 基础偏好 │   │ AI设置   │   │ 第一个   │   │ 开始     │
│          │   │          │   │ (可选)   │   │ 目标/任务│   │ 工作！   │
└──────────┘   └──────────┘   └──────────┘   └──────────┘   └──────────┘
                                                │ [跳过全部] → 直接进入主界面
```

**Step 1 — 欢迎页**: 展示产品核心功能（专注计时/数据洞察/AI助手/温和反馈）,提供"开始设置"和"跳过直接使用"两个入口。

**Step 2 — 基础偏好**: 选择专注时长偏好（25/45/90分钟）、外观（浅色/深色/跟随系统）、日结算时间。

**Step 3 — AI 设置（可选）**: 配置 AI 提供商和 API Key,选择语气风格。包含隐私提示: "你的任务名称和描述会发送到 AI 服务商处理,FocusLab 不会存储或上传数据到我们的服务器"。可跳过。

**Step 4 — 创建第一个目标或任务**: 用户选择创建长线目标（可让AI规划里程碑）或直接添加今天的任务。至少创建1个任务或目标,确保进入主界面时不是空白的。

**Step 5 — 开始工作**: 展示快速指南和快捷键,点击"开始工作"进入主界面。

### 11.3 FTUE 数据持久化

- FTUE 过程中产生的数据直接写入正式数据表（settings/goals/tasks）
- `settings: ftue_completed = true` 标记完成
- 如果用户中途关闭应用,下次启动从上次未完成的步骤继续（记录 `ftue_last_step`）

## 12. 重复任务完整设计（v1.1 新增）

### 12.1 重复规则数据结构

`tasks.recurrence_rule` 字段使用 JSON 格式:

```json
{
  "frequency": "daily | weekly | monthly",
  "interval": 1,
  "days_of_week": [1, 3, 5],
  "day_of_month": 15,
  "end_type": "never | after_count | until_date",
  "end_count": 10,
  "end_date": "2025-12-31"
}
```

**常见场景**: 每天（写日记/锻炼）、每周特定几天（课程）、每两周（导师会面）、每月（月度汇报）

### 12.2 实例生成策略

**按需生成,不预生成所有实例**:
- 触发时机: 每天日期分界线时刻或用户当天首次打开应用
- 检查所有 `is_recurring=true` 且未搁置的任务,判断今天是否为执行日
- 如果是 → 在 `daily_task_assignments` 中创建记录（`source='recurring'`, `is_planned=true`）
- 连续多天没打开应用 → 不补生成过去的实例,只生成今天的
- 搁置某天实例只影响那一天,明天的实例仍正常生成

### 12.3 交互设计

**创建**: 任务创建表单中增加"重复"开关,打开后展示频率选择器（每天/每周/每月）、重复日选择、间隔设置、结束条件。

**列表展示**: 重复任务显示 🔄 图标 + 规则简述（如"每周一三五"）,在每日计划中来源为 recurring 的任务显示浅色底色。

**编辑**: 修改内容影响所有未来实例;修改重复规则需确认;删除某天实例只影响当天;停止重复将任务变为普通任务。

## 13. 任务完成微复盘（v1.1 新增）

### 13.1 触发条件

**必定触发**（弹出微复盘卡片）:
- 任务有预估时长,且实际时长偏差超过 30%
- 任务被标记为"紧急重要"（第一象限）
- 任务关联了长线目标的里程碑

**不触发**（静默完成）:
- 事务性小任务（第三/四象限）
- 时间偏差 < 30%
- 用户关闭了微复盘功能
- 同一天已弹出过3次微复盘

**可选触发**: 任何任务完成时都显示"添加复盘笔记 📝"小入口,用户点击才展开。

### 13.2 微复盘卡片

```
┌─────────────────────────────────────────────────┐
│ 📝 快速复盘                              [✕]   │
│                                                 │
│ 任务：复现baseline代码                          │
│ ⏱️ 预估：2小时  实际：3小时28分钟（超出 74%）   │
│                                                 │
│ 花了更久的原因？（点选，可多选）                │
│ ☐ 📚 比预期难  ☐ 🔔 中途被打断                 │
│ ☐ ➕ 做了额外工作  ☐ 📱 分心了  ☐ 🎯 预估不准  │
│                                                 │
│ 收获/备注（可选）：[                         ]  │
│                                                 │
│ [跳过]                               [保存 ✓]  │
└─────────────────────────────────────────────────┘
```

### 13.3 复盘数据的价值

- **改进时间预估**: 同类任务 ≥5次复盘后,AI 分析预估偏差模式,新任务预估时给出校正建议
- **丰富周报**: 增加"预估准确度"板块,展示偏差趋势
- **科研日志**: 复盘备注可按长线目标→里程碑→时间线导出为 Markdown 科研日志
- **中断模式分析**: 交叉分析复盘原因和中断数据,识别规律（如"周三下午被打断概率最高"）
