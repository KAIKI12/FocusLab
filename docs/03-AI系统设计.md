## 1. AI 能力分层

### Level 1 - 基础能力（低API消耗,高频使用）

- 日结算鼓励话术生成
- 每日建议生成（基于昨日数据）
- 未完成任务的温和提醒文案
- 任务完成后的正反馈文案
- 预估API消耗: 每用户每天 3-5 次调用

### Level 2 - 分析能力（中等API消耗,中频使用）

- 任务智能拆解（大任务 → 子任务列表）
- 四象限自动分类建议
- 周报/月报文字总结生成
- 时间分配分析与建议
- 效率趋势分析
- 任务预估时长建议（基于历史数据）
- 预估API消耗: 每用户每天 2-3 次调用

### Level 3 - 规划能力（高API消耗,低频使用）

- 基于历史数据的每日计划建议
- 长线任务风险预警（"按当前进度可能延期"）
- 里程碑拆解建议（用户输入大目标,AI建议里程碑划分）
- 个性化效率提升方案
- 预估API消耗: 每用户每周 3-5 次调用

## 2. AI 有效性边界

### AI 能做好的事（数据驱动,客观）

- ✅ 呈现数据事实: "你今天在XX上花了3小时"
- ✅ 识别行为模式: "你通常上午效率最高"
- ✅ 任务拆解: "把'写论文'拆成具体步骤"
- ✅ 时间分析: "本周科研时间占比下降了10%"
- ✅ 生成鼓励文案: 基于完成情况的个性化鼓励

### AI 做不好的事（需要领域知识,主观）

- ❌ 判断科研质量: "你的论文写得好不好"
- ❌ 判断任务难度: "这篇论文应该很容易读完"
- ❌ 替代导师指导: "你应该用XX方法做研究"
- ❌ 判断效率原因: "你今天效率低是因为..."
- ❌ 预测精确时间: "你的论文还需要23天完成"

### 应对策略

- AI 只在能力边界内给建议
- 涉及学术判断时,AI 明确说"这需要和导师讨论"
- 所有AI建议都标注"仅供参考"
- 用户反馈机制: 每条AI建议都有 👍👎 按钮
- 基于用户反馈持续优化prompt

## 3. Prompt 模板系统

后端预设多套 Prompt 模板,根据场景和用户设置动态组合。

### 3.1 日结算话术模板

**S级模板**:
```
system: "你是一个{tone}的AI助手,正在为用户进行每日工作结算。"
user: "用户今天的数据如下：
* 完成率：{completion_rate}%
* 已完成任务：{completed_tasks}
* 额外完成：{extra_tasks}
* 总专注时长：{focus_time}
* 专注最久的任务：{longest_focus_task}
请生成一段50字以内的结算寄语,要求：
1. 肯定用户的超额完成
2. 提及具体的亮点数据
3. 对明天给出简短展望
4. 语气{tone}"
```

**C级模板**:
```
system: "你是一个{tone}的AI助手,正在为用户进行每日工作结算。"
user: "用户今天的数据如下：
* 完成率：{completion_rate}%
* 已完成任务：{completed_tasks}
* 未完成任务：{uncompleted_tasks}
* 总专注时长：{focus_time}
请生成一段50字以内的结算寄语,要求：
1. 不要使用'失败''落后''拖延'等负面词汇
2. 肯定用户已经完成的部分
3. 对未完成的部分表示理解
4. 给出温和的调整建议
5. 语气{tone}"
```

**模板变量**: `{completion_rate}`, `{completed_tasks}`, `{focus_time}`, `{longest_focus_task}`, `{uncompleted_tasks}`, `{grade}`, `{tone}`

### 3.2 任务拆解模板

```
system: "你是一个任务规划助手,帮助研究生将大任务拆解为可执行的子任务。"
user: "请将以下任务拆解为3-7个子任务：
任务名称：{task_name}
任务描述：{task_description}
预估总时长：{estimated_time}
关联长线目标：{related_goal}
要求：
1. 每个子任务应该可以在一个番茄钟（25-45分钟）内完成
2. 子任务之间有清晰的先后顺序
3. 为每个子任务标注预估时长
4. 为每个子任务建议四象限分类
5. 输出JSON格式"
```

**模板变量**: `{task_name}`, `{task_description}`, `{estimated_time}`, `{related_goal}`, `{user_history}`

### 3.3 周报总结模板

**模板变量**: `{weekly_stats}`, `{daily_breakdown}`, `{goal_progress}`, `{time_distribution}`, `{comparison_with_last_week}`

**生成内容**: 文字总结 + 改进建议 + 下周展望

### 3.4 引导思考建议模板

**模板变量**: `{yesterday_summary}`, `{goal_progress}`, `{pending_tasks}`, `{user_energy_level}`

**生成内容**: 今日推荐任务列表 + 优先级建议

## 4. AI 语气风格系统

用户可在设置中选择AI的交互风格:

### 预设风格

| 风格 | 特点 | 示例 | 适合用户 |
|------|------|------|---------|
| 📚 学术导师型 | 专业、理性、有条理 | "根据本周数据,你在文献阅读上的时间投入充足,建议下周将重心转向方法论设计。" | 喜欢理性分析 |
| 💪 健身教练型 | 热情、激励、直接 | "今天干得漂亮！7小时专注,这就是冠军的节奏！明天继续冲！" | 需要外部激励 |
| 🧘 禅意陪伴型 | 温和、包容、不施压 | "今天已经很努力了。完成了3个任务,这就是实实在在的进步。好好休息吧。" | 容易焦虑 |
| 🤖 极简数据型 | 简洁、只说数据、不带情感 | "完成率68%。建议明天优先处理遗留的2个任务。" | 不喜欢废话 |
| 😸 猫咪陪伴型 | 可爱、轻松、减压 | "喵～今天完成了5个任务呢,主人好厉害！明天也要加油喵～" | 喜欢轻松氛围 |
| 🎨 自定义风格 | 用户自己编写prompt描述 | 例: "请用东北话风格给我做总结" | 有prompt经验 |

**语气强度滑块**:
```
[低] ────────●──── [高]
↑                  ↑
少说话/简洁      多鼓励/详细
```

**技术实现**:
- 每种风格对应一个 system prompt 模板
- 语气强度影响生成文本的长度和情感浓度
- 用户切换风格后,下次AI生成即生效
- 风格设置存储在本地 settings 表中

## 5. AI 调用策略与成本控制

### 5.1 成本控制策略

**1. 本地缓存**
- 常用的鼓励话术预生成一批,存在本地
- 简单场景（如"任务完成"）直接用本地话术,不调API
- 只有需要个性化分析时才调用API
- 预估可减少 40% 的API调用

**2. 批量调用**
- 日结算时一次性生成所有需要的文案
- 而不是每完成一个任务就调一次API

**3. 分级调用**
- Level 1 功能: 优先使用本地模板 + 变量替换
- Level 2 功能: 使用轻量模型（如 GPT-4o-mini）
- Level 3 功能: 使用完整模型（如 GPT-4o / Claude）
- 用户可在设置中选择AI质量等级（影响API消耗）

**4. API Key 管理**
- MVP阶段: 用户自备API Key（OpenAI / Claude / 其他）
- 后期: 提供官方API Key（付费订阅模式）
- 支持多个API提供商（OpenAI、Anthropic、本地模型）
- API Key 加密存储在本地

**5. 离线降级**
- 无网络时,AI功能降级为本地预设话术
- 任务拆解降级为手动拆解
- 核心功能（计时、任务管理）完全离线可用
- 网络恢复后自动同步需要AI处理的内容

### 5.2 预估月度成本（单用户）

| 使用程度 | 功能范围 | 预估成本 |
|---------|---------|---------|
| 轻度使用 | 仅日结算话术 | 约 $0.5/月 |
| 中度使用 | + 任务拆解 + 周报 | 约 $2/月 |
| 重度使用 | 全部AI功能 | 约 $5/月 |

使用 GPT-4o-mini 可降低约 60% 成本。

## 6. AI 幻觉防护与输出校验

Prompt 模板保证了输入质量，但 AI 仍可能产生不合理输出（无意义子任务、不当话术、格式错误等）。本节定义输出侧的防护机制。

### 6.1 输出格式校验

- AI 拆解结果要求返回 JSON，前端解析失败时显示"AI 拆解失败，请手动创建"
- 校验子任务数量（3-7 个）、预估时间（5-480 分钟）
- 不合理的值自动修正或丢弃

**校验规则示例**:
```rust
if subtask.estimated_minutes < 5   → 设为 15
if subtask.estimated_minutes > 480 → 设为 120
if subtasks.len() > 10             → 只取前 7 个
if subtasks.len() < 1              → 标记为拆解失败
```

### 6.2 文案内容过滤

- 日结算话术生成后，检查是否包含禁用词
  - 禁用词列表：失败、落后、拖延、差劲、糟糕、废物……
  - 命中禁用词 → 替换为预设的安全话术
  - 记录到 `ai_feedbacks` 表，标记为 `filtered`
- 检查话术长度：
  - < 10 字 → 可能是无效输出，用预设话术替代
  - \> 200 字 → 截断到 150 字
- 在 Prompt 中加入输出约束（已部分体现，需持续加强）

### 6.3 超时与重试

- AI 调用超时：**30 秒**
- 超时后：立即降级为本地预设话术
- 重试策略：**不自动重试**（避免重复消耗 token）
- 用户可手动点击"重新生成"
- 连续失败 3 次 → 提示"AI 服务暂时不可用，请检查网络或 API Key"

### 6.4 成本异常检测

- 单次调用 token > 2000 → 警告
- 单日总 token > 50000 → 暂停 AI 功能并通知用户
- 防止 Prompt 注入导致的异常消耗
- 在设置中显示"本月 AI 使用量"概览

## 7. AI 上下文窗口管理策略

随着用户使用时间增长，可用上下文越来越多。发送太多会浪费 token、拖慢响应甚至超出模型窗口；发送太少则影响建议质量。本节为每个 AI 功能定义精确的上下文范围。

### 7.1 上下文策略表

| 功能 | 发送的上下文 | 预估 token |
|------|-------------|-----------|
| 日结算话术 | 当天统计数据（数字摘要） | 200-300 |
| 每日建议 | 昨日摘要 + 当前里程碑状态 + 遗留任务 | 500-800 |
| 任务拆解 | 任务名 + 描述 + 关联目标名称 | 300-500 |
| 周报总结 | 7 天统计数据聚合 + 目标进度摘要 | 800-1200 |
| 四象限分类 | 任务名 + 描述 + 现有任务的象限分布 | 300-500 |

### 7.2 关键原则

- **只发送聚合后的统计数据，不发送原始记录**
  - ✅ "本周科研时间 18 小时，事务性时间 6 小时"
  - ❌ 发送这一周所有 200 条 session 记录
- **任务名称和描述只发送当前相关的**
  - 任务拆解时只发送要拆解的那个任务，不发送用户所有历史任务
- **长线目标只发送名称和里程碑状态**，不发送全部子任务
- **后端在调用 AI 前负责组装精简上下文**（`services/ai_context_builder.rs`）
- **每次调用前记录实际发送的 token 数**，用于成本统计

## 8. 多 AI 提供商适配层

文档已提到支持 OpenAI / Anthropic / 其他兼容 API，本节补充统一的适配层设计，使业务代码与具体提供商解耦。

### 8.1 AIProvider trait 设计（Rust）

```rust
// ai/provider.rs

trait AIProvider {
    async fn complete(
        &self,
        messages: Vec<Message>,
        options: CompletionOptions,
    ) -> Result<String, AIError>;

    async fn complete_stream(
        &self,
        messages: Vec<Message>,
        options: CompletionOptions,
    ) -> Result<impl Stream<Item = Result<String, AIError>>, AIError>;
}

struct OpenAIProvider      { api_key: String, base_url: String, model: String }
struct AnthropicProvider    { api_key: String, model: String }
struct OllamaProvider      { base_url: String, model: String }  // 本地模型
struct CompatibleProvider   { api_key: String, base_url: String, model: String }
    // 兼容 OpenAI 格式的第三方 API（如 DeepSeek、智谱等）
```

### 8.2 工厂函数

```rust
fn create_provider(settings: &Settings) -> Box<dyn AIProvider> {
    match settings.ai_provider.as_str() {
        "openai"     => Box::new(OpenAIProvider::new(...)),
        "anthropic"  => Box::new(AnthropicProvider::new(...)),
        "ollama"     => Box::new(OllamaProvider::new(...)),
        "compatible" => Box::new(CompatibleProvider::new(...)),
        _            => panic!("Unknown provider"),
    }
}
```

### 8.3 设置界面支持

- 提供商选择：OpenAI / Anthropic / 本地模型(Ollama) / 自定义兼容 API
- 自定义兼容 API 需要填写：
  - API 地址（Base URL）
  - API Key
  - 模型名称
  - [测试连接] 按钮
- 可支持的第三方服务：
  - DeepSeek API
  - 智谱 ChatGLM API
  - Cloudflare Workers AI
  - 各种 OpenAI 兼容代理
  - 自建 API 网关
- 对中国用户特别重要（OpenAI 直连困难）

### 8.4 settings 表新增字段

| 字段 | 类型 | 示例值 | 说明 |
|------|------|--------|------|
| `ai_provider` | TEXT | `"openai"` / `"anthropic"` / `"ollama"` / `"compatible"` | 提供商标识 |
| `ai_base_url` | TEXT | `"https://api.openai.com/v1"` | API 地址，可自定义 |
| `ai_model` | TEXT | `"gpt-4o-mini"` | 模型名称 |
| `ai_api_key` | TEXT | — | 加密存储 |
